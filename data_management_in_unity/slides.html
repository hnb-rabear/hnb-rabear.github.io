Bối cảnh

Một mobile game mid-core điển hình có bao nhiêu data?

200+

config fields cần balance
(item stats, level curves, rewards...)

15+

player data models cần persist
(profile, inventory, quests, settings...)

3-5

team members cùng sửa data layer
mỗi sprint

Observation

Không có data architecture rõ ràng → phần lớn bugs trong game không phải logic sai,
mà là data sai chỗ, save sai lúc, hoặc đọc sai state.

Quá trình tiến hóa

3 giai đoạn mà đa số team đều trải qua

Gen 1

PlayerPrefs trực tiếp

SetInt/GetString rải khắp nơi. Config hardcode trong code. Ship nhanh nhưng 3 tháng sau không maintain được.

Gen 2

Centralized + JSON

1 class SaveManager. Config từ ScriptableObject. Tốt hơn nhưng vẫn thiếu lifecycle management và validation.

Gen 3

Layered Architecture

Tách Data / Logic / Lifecycle. Config pipeline từ Sheets. Auto-save. Event-driven. Test được.

Key realization

Bước nhảy lớn nhất không phải Gen 1 → Gen 2 (thêm code). Mà là Gen 2 → Gen 3 (thay đổi tư duy thiết kế).

3 nguyên tắc cốt lõi

Mọi quyết định thiết kế đều quay về 3 điều này

01

Tách biệt mối quan tâm

Data structure ≠ Business logic ≠ Persistence.

Designer quan tâm "item có giá bao nhiêu". Dev quan tâm "validate trước khi trừ coin". Framework quan tâm "save khi nào".

02

Một cổng duy nhất

Mọi thay đổi data phải đi qua đúng 1 nơi — "người gác cổng".

Nếu 5 chỗ trong code đều sửa được coin trực tiếp → không ai kiểm soát coin có hợp lệ không.

03

Không phụ thuộc ngầm

Module A thay đổi data → Module B tự biết và tự xử lý, không cần A gọi trực tiếp B.

Event-driven. Thêm module mới = subscribe. Xóa module = unsubscribe. Zero side effects.

Hai trụ cột của data architecture

Read-Only

Static Config Data

Dữ liệu được Designer quản lý: item stats, level curves, rewards, localization.

Spreadsheet → Export tool → JSON + C# IDs → ScriptableObject

Thay đổi khi nào? Mỗi build mới. Hoặc qua Remote Config.

Read / Write

Dynamic Player Data

Dữ liệu runtime: player profile, inventory, quest progress, settings.

Data Model → Data Handler → Collection → Persistence

Thay đổi khi nào? Mỗi action của player. Auto-save on pause/quit.

Quy tắc phân loại

Nếu Designer sửa nó → Static Config. Nếu player tạo ra nó → Dynamic Data. Không có loại thứ 3.

Pattern: Config Pipeline

Một config pipeline tốt loại bỏ dev ra khỏi vòng lặp balance

Config class — auto-generated từ sheet schema
[Serializable]
public class ShopItemConfig
{
    public int id;
    public string name;
    public int price;
    public int currencyType;
}
Truy xuất — type-safe, compile-time checked
// Gõ sai ID? Compile error. Không đợi runtime.
var item = configCollection.GetShopItem(IDs.Shop.COIN_PACK_1);
var price = item.price;

Vì sao pattern này quan trọng?

  • Designer sửa giá item: 5 phút (trước: 2 ngày)
  • Thêm item mới: thêm row trong sheet, không sửa code
  • Gõ sai tên ID: compile error ngay, không phải runtime null
  • Ai đã sửa gì: Git history của sheet

Lưu ý từ production

Dùng partial class để chia ConfigCollection theo feature. 1 file chung + N file riêng. Giảm merge conflict khi nhiều dev cùng thêm config.

Pattern: Repository — "Người gác cổng"

Tách data structure ra khỏi business logic — nguyên tắc #1 trong thực hành

Layer 1: Data Model (DTO) — chỉ chứa data
[Serializable]
public partial class PlayerData
{
    public int coin;
    public int level;
    public List<int> ownedItems = new();
    // Không có method. Không có logic.
    // Serializable. Portable. Testable.
}
Layer 2: Data Handler — chứa logic
public class PlayerModel : DataHandler<PlayerData>
{
    public event Action<int> OnCoinChanged;

    public bool TrySpendCoin(int amount)
    {
        if (Data.coin < amount) return false;
        Data.coin -= amount;
        OnCoinChanged?.Invoke(Data.coin);
        MarkDirty();  // Đánh dấu cần save
        return true;
    }
}
Tại sao tách?

Data Model cần serialize → JSON, gửi server, lưu disk. Nếu chứa logic → serialize dính logic → phức tạp.
Data Handler là "người gác cổng" — validate, notify, track dirty. 1 nơi duy nhất sửa data = 0 bug "ai đã sửa coin?".

Pattern: Lifecycle Management

Data không chỉ load/save. Nó có vòng đời phức tạp hơn nhiều.

Init()

Lần đầu chơi → set default values, tutorial flags

OnPostLoad(utcNow, offlineSeconds)

Load xong → tính offline rewards, regenerate lives, check expired timers

OnUpdate(deltaTime)

Mỗi frame → countdown timers, regen resources, check conditions

OnPause() → OnPreSave()

App vào background → validate + auto-save. Không bao giờ mất data.

Ví dụ: Offline reward calculation
public override void OnPostLoad(
    int utcNow, int offlineSeconds)
{
    // Player rời game 3 giờ
    // → tặng gold dựa trên thời gian offline
    int goldPerMin = ConfigCollection
        .Instance.offlineGoldRate;
    int reward = offlineSeconds / 60 * goldPerMin;
    AddCoin(reward);

    // Regenerate lives
    int livesRegen = offlineSeconds
        / LIVE_REGEN_SECONDS;
    Data.lives = Math.Min(
        Data.lives + livesRegen, MAX_LIVES);
}

Framework gọi đúng hook, đúng thời điểm. Dev chỉ viết logic.

Pattern: Observer — Decoupled Communication

Nguyên tắc #3: modules không phụ thuộc ngầm vào nhau

❌ Cách sai — tight coupling

void SpendCoin(int amount) {
    coin -= amount;
    hudView.UpdateCoin(coin);    // biết HUD
    shopView.Refresh();          // biết Shop
    questMgr.CheckProgress();    // biết Quest
    // Thêm feature = sửa file này
    // 10 features = 10 dependencies
}

PlayerModel phải biết tất cả consumers. Thêm 1 feature = sửa PlayerModel.

✅ Cách đúng — event-driven

void SpendCoin(int amount) {
    Data.coin -= amount;
    OnCoinChanged?.Invoke(Data.coin);
    MarkDirty();
    // Thêm feature? Subscribe event.
    // Xóa feature? Unsubscribe.
    // PlayerModel không bao giờ sửa.
}

PlayerModel chỉ phát tín hiệu. Ai quan tâm tự lắng nghe. Zero coupling.

Kinh nghiệm thực tế

Khi project scale lên 15+ features, tight coupling khiến mỗi lần thêm feature mới phải sửa 3-4 file cũ.
Event-driven: thêm feature = 1 file mới, 0 file cũ bị sửa.

Scaling: Khi project lớn lên

3 kỹ thuật giữ cho data layer không trở thành "big ball of mud"

partial class

// PlayerData.cs
// PlayerData.Inventory.cs
// PlayerData.Avatar.cs

// ConfigCollection.cs
// ConfigCollection.Blitz.cs

3 dev cùng sửa PlayerData → mỗi người 1 file → 0 merge conflict.

Module boundaries

// Thêm feature Quest:
// 1. QuestData.cs    (data)
// 2. QuestsModel.cs  (logic)
// 3. SaveDataCollection:
[AutoFill] public QuestsModel quests;
// Xong. Không sửa code cũ.

Open/Closed: mở rộng bằng thêm mới, không bằng sửa cũ.

Cross-model access

class PlayerModel : DataHandler
{
    [SerializeField]
    CompetitionModel m_competition;

    // ✅ ĐỌC dữ liệu model khác
    // ❌ KHÔNG SỬA dữ liệu model khác
    // Sửa → dùng Events
}

Một chiều: đọc trực tiếp, sửa qua event.

Những sai lầm đã trả giá

Kinh nghiệm không đến từ thành công — mà từ debug sessions lúc 2 giờ sáng

"Đặt logic trong Data Model"
6 tháng sau: 15 chỗ gọi playerData.AddCoin() trực tiếp. Không ai validate. Không ai notify UI. Phải refactor toàn bộ.
"1 file PlayerData.cs — 2000 dòng"
Serialize toàn bộ mỗi lần save = 50ms lag spike trên low-end device. Merge conflict mỗi ngày. Chia partial class = hết cả hai.
"Thêm field mới, không check version"
Player cũ update app → field mới = default 0. Nhưng business cần default = 100 (quà chào mừng). Bỏ lỡ 50K users. Giờ luôn check dataVersion trong OnPostLoad().
"Quên test trên low-end device"
PlayerPrefs write trên flagship: 2ms. Trên budget phone 2019: 80ms. Main thread. Frame drop rõ rệt. Solution: MarkDirty() + batch save on pause.

Trade-offs chúng tôi chấp nhận

Mọi kiến trúc đều có giá. Quan trọng là biết mình đang trả cái gì.

Chúng tôi được

  • Designer tự sửa balance — không cần Dev
  • Auto-save — 0 reports mất progress
  • Type-safe IDs — compile error > runtime crash
  • Modular — thêm feature mà không sửa code cũ
  • Inspector-friendly — debug trực tiếp trong Editor
  • Proven — chạy ổn trên production apps

Chúng tôi chấp nhận

  • Local storage không encrypt by default → cần thêm layer
  • Không có cloud save built-in → integrate riêng
  • Singleton access → khó mock trong unit tests
  • Config baked trong build → cần Remote Config cho hotfix
  • Framework-dependent → migration cost nếu đổi stack
Khi nào kiến trúc này KHÔNG phù hợp?

Multiplayer cần server-authoritative data. AAA cần async file I/O. Nếu bạn đang làm game online competitive → cần thêm server validation layer.
