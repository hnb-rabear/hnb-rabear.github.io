Observation

Không có data architecture rõ ràng → phần lớn bugs không phải logic sai, mà là data sai chỗ, save sai lúc, hoặc đọc sai state.

Quá trình tiến hóa

3 giai đoạn mà đa số team đều trải qua

Gen 1

PlayerPrefs trực tiếp

SetInt/GetString rải khắp nơi. Config hardcode. Ship nhanh nhưng 3 tháng sau không maintain được.

Gen 2

Centralized + JSON

1 class SaveManager, config từ ScriptableObject. Tốt hơn nhưng thiếu lifecycle management.

Gen 3

Layered Architecture

Tách Data / Logic / Lifecycle. Config pipeline từ Sheets. Auto-save. Event-driven.

Key realization

Bước nhảy lớn nhất không phải Gen 1 → Gen 2 (thêm code). Mà là Gen 2 → Gen 3 (thay đổi tư duy thiết kế).

3 nguyên tắc cốt lõi

Mọi quyết định thiết kế đều quay về 3 điều này

01

Tách biệt mối quan tâm

Data structure ≠ Business logic ≠ Persistence.

Ai quan tâm gì, người đó xử lý.

02

Một cổng duy nhất

Mọi thay đổi data đi qua 1 nơi — "người gác cổng".

5 chỗ sửa coin = không ai kiểm soát.

03

Không phụ thuộc ngầm

Module A thay đổi → phát event. Module B tự lắng nghe.

Thêm = subscribe. Xóa = unsubscribe.

Hai trụ cột của data architecture

Read-Only

Static Config Data

Dữ liệu được Designer quản lý: item stats, level curves, rewards, localization.

Spreadsheet → SheetX → JSON + C# → ScriptableObject

Thay đổi khi nào? Mỗi build mới hoặc Remote Config.

Read / Write

Dynamic Player Data

Dữ liệu runtime: player profile, inventory, quest progress, settings.

Data Model → Handler → Collection → Persistence

Thay đổi khi nào? Mỗi action của player. Auto-save on pause.

Pattern: Config Pipeline

Loại bỏ dev ra khỏi vòng lặp balance

Config class — tương ứng sheet schema
[Serializable]
public class ShopItemConfig
{
    public int id;
    public string name;
    public int price;
    public int currencyType;
}
Truy xuất — type-safe, compile-time checked
// Gõ sai ID? Compile error.
var item = config.GetShopItem(IDs.Shop.COIN_PACK_1);
var price = item.price;

Vì sao pattern này quan trọng?

  • Designer sửa giá: 5 phút (trước: 2 ngày)
  • Thêm item: thêm row, không sửa code
  • Gõ sai ID: compile error ngay

Tip từ production

Dùng partial class chia ConfigCollection theo feature → giảm merge conflict khi nhiều dev cùng thêm config.

Pattern: Repository — "Người gác cổng"

Tách data structure ra khỏi business logic

Layer 1: Data Model — chỉ chứa data
[Serializable]
public partial class PlayerData
{
    public int coin;
    public int level;
    public List<int> ownedItems = new();
    // Không method. Không logic.
}
Layer 2: Data Handler — chứa logic
public class PlayerModel : DataHandler<PlayerData>
{
    public event Action<int> OnCoinChanged;

    public bool TrySpendCoin(int amount)
    {
        if (Data.coin < amount) return false;
        Data.coin -= amount;
        OnCoinChanged?.Invoke(Data.coin);
        MarkDirty();  // Đánh dấu cần save
        return true;
    }
}
Tại sao tách?

Data Model cần serialize (JSON, gửi server). Data Handler là "người gác cổng" — validate, notify, track dirty. 1 nơi sửa data = 0 bug "ai đã sửa coin?".

Pattern: Lifecycle Management

Data có vòng đời phức tạp hơn chỉ load/save

Init()

Lần đầu chơi → set default values

OnPostLoad(utcNow, offlineSecs)

Load xong → tính offline rewards, check expired timers

OnUpdate(deltaTime)

Mỗi frame → countdown timers, regen resources

OnPause() → OnPreSave()

App background → validate + auto-save

Ví dụ: Offline reward
public override void OnPostLoad(
    int utcNow, int offlineSeconds)
{
    int goldPerMin = ConfigCollection
        .Instance.offlineGoldRate;
    int reward = offlineSeconds / 60
        * goldPerMin;
    AddCoin(reward);

    int livesRegen = offlineSeconds
        / LIVE_REGEN_SECONDS;
    Data.lives = Math.Min(
        Data.lives + livesRegen, MAX_LIVES);
}

Framework gọi đúng hook, đúng thời điểm. Dev chỉ viết logic.

Pattern: Observer — Decoupled Communication

Nguyên tắc #3: modules không phụ thuộc ngầm vào nhau

❌ Tight coupling

void SpendCoin(int amount) {
    coin -= amount;
    hudView.UpdateCoin(coin);    // biết HUD
    shopView.Refresh();          // biết Shop
    questMgr.CheckProgress();    // biết Quest
    // 10 features = 10 dependencies
}

Thêm feature = sửa PlayerModel.

✅ Event-driven

void SpendCoin(int amount) {
    Data.coin -= amount;
    OnCoinChanged?.Invoke(Data.coin);
    MarkDirty();
    // Thêm feature? Subscribe event.
    // Xóa feature? Unsubscribe.
}

PlayerModel chỉ phát tín hiệu. Zero coupling.

Kinh nghiệm thực tế

15+ features với tight coupling: thêm feature = sửa 3-4 file cũ. Event-driven: thêm feature = 1 file mới, 0 file cũ bị sửa.

Những sai lầm đã trả giá

Kinh nghiệm không đến từ thành công — mà từ debug sessions lúc 2 giờ sáng

"Đặt logic trong Data Model"
6 tháng sau: 15 chỗ gọi playerData.AddCoin() trực tiếp. Không validate. Không notify UI. Phải refactor toàn bộ.
"1 file PlayerData.cs — 2000 dòng"
Serialize toàn bộ mỗi lần save = 50ms lag spike trên low-end device. Merge conflict mỗi ngày. partial class = hết cả hai.
"Thêm field mới, không check version"
Player cũ update app → field mới = default 0, business cần 100. Bỏ lỡ 50K users. Giờ luôn check dataVersion trong OnPostLoad().
"Quên test trên low-end device"
PlayerPrefs write trên flagship: 2ms. Budget phone 2019: 80ms main thread. Solution: MarkDirty() + batch save on pause.

Trade-offs chúng tôi chấp nhận

Mọi kiến trúc đều có giá. Quan trọng là biết mình đang trả cái gì.

Chúng tôi được

  • Designer tự sửa balance — không cần Dev
  • Auto-save — 0 reports mất progress
  • Type-safe IDs — compile error > runtime crash
  • Modular — thêm feature không sửa code cũ
  • Inspector-friendly — debug trực tiếp Editor

Chúng tôi chấp nhận

  • Local storage không encrypt by default
  • Không cloud save built-in
  • Singleton → khó mock trong unit tests
  • Config baked → cần Remote Config cho hotfix
  • Framework-dependent → migration cost
Khi nào KHÔNG phù hợp?

Multiplayer cần server-authoritative data. AAA cần async file I/O. Game online competitive → cần thêm server validation layer.

Editor Tools — Điều tạo nên sự khác biệt

Xem và chỉnh sửa player data trực tiếp trong Unity Editor, không cần build

JObjectDB Editor — View JSON data toàn bộ hệ thống
ScriptableObject Inspector — Edit từng field trực tiếp
Tác động thực tế

Debug speed tăng 5-10x — thay vì print log → build → reproduce → check, bạn mở Inspector → thấy ngay state → sửa → test lại trong 3 giây.
