Quá trình tiến hóa

3 giai đoạn mà đa số dev đều trải qua

Gen 1: Spaghetti Data

PlayerPrefs tùy tiện

Set/GetInt rải khắp nơi. Hardcode config. Đổi 1 chỗ → vỡ 5 chỗ.

Gen 2: God Class

Central SaveManager

1 Singleton gom hết. Gọn hơn, nhưng God Class 3000 dòng + dễ Merge Conflict.

Gen 3: Decoupled

Layered System

Tách Data - Logic - Persistence. Chia theo Domain. Event-driven.

Key realization

Gen 1 → 2: gom rác về một chỗ (Centralize). Gen 2 → 3: chia nhỏ có hệ thống (Decouple).

3 nguyên tắc cốt lõi

Mọi quyết định thiết kế đều quay về 3 điều này

01

Separation of Concerns

Data Model ≠ Game Logic ≠ Local Save.

Tách rõ ranh giới: Data chỉ lưu trạng thái, Handler xử lý luật. Không code gộp.

02

Single Point of Access

Mọi lệnh Write data bắt buộc phải đi qua "người gác cổng" để Validate.

5 feature cùng tự ý trừ Coin = vỡ trận state!

03

Event-Driven

Các module không nhìn thấy nhau. Logic đổi state → phát Event.

Khả năng Plug & Play: Thêm tính năng = Subscribe, Xóa = Unsubscribe.

Pattern: Config Pipeline

Loại bỏ dev ra khỏi vòng lặp balance

Config class — tương ứng sheet schema
[Serializable]
public class ShopItemConfig
{
    public int id;
    public string name;
    public int price;
    public int currencyType;
}
Truy xuất — type-safe, compile-time checked
// Gõ sai ID? Compile error.
var item = config.GetShopItem(IDs.Shop.COIN_PACK_1);
var price = item.price;

Vì sao pattern này quan trọng?

  • Sửa giá trị: Designer tự làm, không cần Dev
  • Thêm field mới: Dev chỉ phải bổ sung field mới
  • ID auto-generated → dev không hardcode string

🔧 Tool: SheetX

Pipeline trên được tự động hóa bởi SheetX — Unity Editor tool xử lý toàn bộ: Google Sheets/Excel → JSON + C# IDs + ScriptableObject.

Pattern: Repository — "Người gác cổng"

Tách data structure ra khỏi business logic — ví dụ từ JObjectDB System

Layer 1: Data Model — chỉ chứa data
[Serializable]
public partial class PlayerData
    : JObjectData
{
    public int level = 1;
    public int coin;
    public int live = 5;
    public int liveRegenTime;
    public List<int> iapIdsBought = new();
    public bool noAds;
}
Layer 2: Data Handler — chứa logic
public partial class PlayerModel
    : JObjectModel<PlayerData>
{
    public void SetCurrency(
        IDs.Currency currency,
        int pValue, string pPlacement)
    {
        // validate + assign...
        DispatchEvent(
            new CurrencyChangedEvent(
                currency, preValue,
                pValue, pPlacement));
    }
}
Tại sao tách?

Data Model cần serialize (JSON, gửi server). Data Handler là "người gác cổng" — validate, notify, track dirty. 1 nơi sửa data = 0 bug "ai đã sửa coin?".

Pattern: Lifecycle Management

Data có vòng đời phức tạp hơn chỉ load/save

Init()

Lần đầu chơi → set default values

OnPostLoad(utcNow, offlineSecs)

Load xong → tính offline rewards, check expired

OnUpdate(deltaTime)

Mỗi frame → countdown timers, regen

OnPause() → OnPreSave()

App background → validate + auto-save

OnRemoteConfigFetched()

Remote config thay đổi → cập nhật data theo config mới

Ví dụ thực tế: Live regen khi offline
public override void OnPostLoad(
    int utcNowTimestamp, int offlineSeconds)
{
    ValidateAvatars();
    data.ValidateLiveModifiers();
    if (offlineSeconds > 0)
        ProcessOfflineRegen(offlineSeconds);
}

Framework gọi đúng hook, đúng thời điểm. Dev chỉ viết logic.

Pattern: Observer — Decoupled Communication

Nguyên tắc #3: Event-Driven — modules không phụ thuộc ngầm vào nhau

❌ Tight coupling

void SpendCoin(int amount) {
    coin -= amount;
    hudView.UpdateCoin(coin);    // biết HUD
    shopView.Refresh();          // biết Shop
    questMgr.CheckProgress();    // biết Quest
    // 10 features = 10 dependencies
}

Thêm feature = sửa PlayerModel.

✅ Event-driven

public void SetCurrency(...) {
    // validate + assign
    DispatchEvent(
        new CurrencyChangedEvent(
            currency, preValue,
            pValue, pPlacement));
}
// UI subscribe CurrencyChangedEvent
// Quest subscribe CurrencyChangedEvent

PlayerModel chỉ phát event. Zero coupling.

Kinh nghiệm thực tế

Code dính chặt: thêm feature mới phải sửa các module không liên quan (PlayerModel, HUD, Quest...). Event-driven: các module cũ không cần biết feature mới tồn tại.

Anti-patterns phổ biến

Những lỗi thực tế từ production — không chỉ junior mà cả dev lâu năm code ẩu cũng mắc

❌ Trộn runtime state với persistent state 1 / 5
SAI
[Serializable]
public class PlayerData : JObjectData
{
    public int coin;
    public int level;
    public float tweenProgress;  // ← animation
    public bool isDragging;      // ← input state
}
ĐÚNG
[Serializable]
public class PlayerData : JObjectData
{
    public int coin;
    public int level;
}
// tweenProgress, isDragging
// → để trong MonoBehaviour (scene lifetime)
❌ Fat Data Model — "God Object" 2 / 5
SAI — 1 file 500+ dòng
// PlayerData.cs — MỌI THỨ ở đây
public class PlayerData : JObjectData
{
    public int coin, level, live;
    public List<int> unlockedAvatars;
    public List<Booster> boosters;
    public int questProgress, raceRank;
    // ... 50+ fields, 2 devs sửa = conflict
}
ĐÚNG — partial class theo domain
// PlayerData.cs — core
public partial class PlayerData : JObjectData
{  public int coin, level, live; }

// PlayerData.Inventory.cs
public partial class PlayerData
{  public List<Booster> boosters; }

// PlayerData.Avatar.cs
public partial class PlayerData
{  public List<int> unlockedAvatars; }
❌ Config data bị mutate runtime 3 / 5
SAI — sửa config từ SheetX trực tiếp
// Config từ Google Sheets (Read-Only!)
var hero = DataConfigCollection
    .Instance.GetHero(heroId);

// Buff bằng cách sửa thẳng config:
hero.attack += buffValue;
hero.maxHp += 200;
// → Config sai vĩnh viễn trong session!
ĐÚNG — tách runtime modifier riêng
// Config giữ nguyên (Read-Only)
var baseAtk = config.GetHero(heroId).attack;

// Runtime modifier tách riêng:
int finalAtk = baseAtk + GetBuffTotal();

// Hoặc dùng LiveModifier pattern:
playerData.AddLiveModifier(id, value, endTime);
❌ Cross-model mutation — bypass "người gác cổng" 4 / 5
SAI — StoreModel sửa data của PlayerModel
// Trong StoreModel — mua item
var player = SaveDataCollection
    .Instance.player;

player.data.CurrentCoins -= item.price;
player.data.iapIdsBought.Add(item.id);
// Bypass PlayerModel validation hoàn toàn!
ĐÚNG — gọi qua PlayerModel methods
// Trong StoreModel — mua item
var player = SaveDataCollection
    .Instance.player;

player.data.AddCurrency(
    IDs.Currency.c_Coin, -price,
    groupPlacement, placement);
// → validate + event + tracking
❌ ScriptableObject gọi MonoBehaviour 5 / 5
SAI — SO phụ thuộc scene
// Trong PlayerModel (ScriptableObject)
var hud = FindObjectOfType<HUDView>();
hud.UpdateCoin(data.coin);

// Scene unload → hud = null → CRASH
// SO tồn tại xuyên scene, MB thì không!
ĐÚNG — phát event, MB tự lắng nghe
// Trong PlayerModel (ScriptableObject)
DispatchEvent(new CurrencyChangedEvent(
    currency, preValue, newValue, placement));

// HUDView (MB) tự subscribe event
// Scene unload → tự unsubscribe → an toàn

Tại sao kiến trúc này hiệu quả?

Những vấn đề thực tế được giải quyết

Single Source of Truth

Mỗi data chỉ tồn tại 1 nơi. Không còn PlayerPrefs.GetInt() rải khắp codebase.

Predictable Data Flow

Mọi thay đổi đều qua Handler → dễ trace, dễ debug. "Ai sửa coin?" — luôn trả lời được.

Zero Coupling giữa Features

Thêm feature = subscribe event. Các module cũ không cần biết feature mới tồn tại.

Giảm lỗi do con người

Auto-save, lifecycle hooks, validation — framework xử lý, dev không cần nhớ.

Team làm việc song song

Partial class + module boundaries → mỗi dev 1 file, 0 merge conflict.

Thay đổi không sợ vỡ

Đổi cách lưu (JSON → encrypt → cloud)? Chỉ sửa persistence layer, game logic không đổi.

Bức tranh lớn hơn: MVP

Data Management là nền tảng của kiến trúc MVP (Model-View-Presenter):
View → Presenter → Data Handler → Data Model
Presenter điều phối logic game, View chỉ hiển thị — cả hai đều không sửa data trực tiếp.
