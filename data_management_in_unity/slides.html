Type-Safe: Lỗi phát hiện ngay khi viết code

// ❌ String — gõ sai chỉ biết khi CHẠY game
var hero = GetHeroById("HEROS_1");  // Runtime: null → crash

// ✅ Generated constant — gõ sai biết NGay khi VIẾT code
var hero = GetHeroById(IDs.Hero.HERO_1);  // Compile-time checked

Tại sao không dùng PlayerPrefs trực tiếp?

❌ PlayerPrefs trực tiếp

// 10 chỗ sửa coin — không ai validate
PlayerPrefs.SetInt("coin", coin - 100);
// Coin có thể âm? Ai notify UI?
// Ai biết cần save? Quên thì mất data.

✅ Data Handler

// 1 chỗ duy nhất — validate + notify + save
public bool TrySpendCoin(int amount) {
    if (Data.coin < amount) return false;
    Data.coin -= amount;
    OnCoinChanged?.Invoke(Data.coin);
    MarkDirty(); // Auto-save khi cần
    return true;
}

1 nơi validate. 1 nơi notify. 1 nơi save. Không bao giờ quên.

Dynamic Data: 4 lớp, 4 trách nhiệm

DBManager (MonoBehaviour) Unity lifecycle · DontDestroyOnLoad · auto-save on pause/quit
SaveDataCollection (ScriptableObject) Tập hợp tất cả Models · điều phối Load/Save/OnUpdate
JObjectModel<T> (ScriptableObject) Logic nghiệp vụ · validation · events · lifecycle hooks
JObjectData (POCO, Serializable) Dữ liệu thuần túy · chỉ chứa fields · không có logic

⚡ Quy tắc vàng

❌  playerData.coin -= 100;              // KHÔNG — sửa data trực tiếp
✅  playerModel.TrySpendCoin(100);       // ĐÚNG  — đi qua "người gác cổng"

Data Handler — chứa logic

public partial class PlayerModel
    : JObjectModel<PlayerData>
{
    public event Action<int> OnCoinChanged;

    public void AddCoin(int amount) {
        Data.coin += amount;
        OnCoinChanged?.Invoke(Data.coin);
        MarkDirty();
    }
}

Lifecycle: Không bao giờ quên save

Bạn chỉ viết logic. Framework lo timing và save.

App Start → Init() hoặc OnPostLoad()

Lần đầu? Set default. Lần sau? Tính offline reward.

Game Loop → OnUpdate(deltaTime)

Mỗi frame: regenerate lives, đếm ngược timer, cooldown...

App Pause → OnPause() + OnPreSave()

Vào background → lưu timestamp → AUTO SAVE

App Quit → OnPreSave() → AUTO SAVE

Đóng app → validate data → save → không bao giờ mất progress

public override void OnPostLoad(int utcNow, int offlineSeconds) {
    int offlineGold = offlineSeconds / 60 * GOLD_PER_MINUTE;
    AddCoin(offlineGold); // Player rời game 3 giờ → +50 gold
}

Ưu điểm & Nhược điểm

Không có kiến trúc hoàn hảo. Đây là trade-offs chúng tôi chấp nhận.

✅ Ưu điểm

  • 🚀 Designer iteration tăng 3-5x
  • 🔧 Unity-native — Inspector, SO, Editor Tools
  • 💾 Auto lifecycle — không quên save
  • 🔒 Type-safe — compile error > runtime crash
  • 🧩 Modular — thêm Data + Model = xong
  • 🔌 Decoupled — events, modules độc lập

❌ Nhược điểm

  • 🔓 PlayerPrefs không encrypt → dễ cheat
  • ☁️ Không có cloud save built-in
  • 📌 Singleton pattern → khó unit test
  • 📦 Config baked trong build → cần Remote Config
  • 🔗 Tight coupling với RCore framework
  • 📏 Không có data versioning built-in

Bài học từ production

"Data Model KHÔNG chứa logic"
Đặt AddCoin() trong PlayerData → 6 tháng sau, 15 chỗ gọi trực tiếp, không ai notify UI.
"Tách data thành nhiều model nhỏ"
1 model PlayerData 2000 dòng → serialize = lag spike trên low-end phone. Chia nhỏ = save nhanh.
"partial class cứu merge conflicts"
3 dev cùng sửa PlayerData.cs = conflict mỗi ngày. Chia thành .cs, .Inventory.cs, .Misc.cs → hết.
"OnPostLoad() là nơi xử lý migration"
Thêm field mới? Default C# = 0. Nhưng player cũ cần nhận 100 gem. Luôn check dataVersion.
"Export IDs TRƯỚC, JSON SAU"
Export JSON trước khi IDs → tham chiếu ID sai giá trị. Thứ tự: IDs → Constants → JSON.
